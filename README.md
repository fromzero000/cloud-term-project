# 프로젝트 제안서: AWS 기반 택시 동승 매칭 서비스

## 1. 프로젝트 개요
* [cite_start]**주제**: AWS 트래픽 분산 처리를 활용한 대학생 맞춤형 택시 동승(합승) 매칭 및 실시간 위치 공유 웹 서비스 [cite: 33, 37]

## 2. 필요성 및 활용방안
* **필요성**:
    * [cite_start]대학생의 경제적 부담 완화 [cite: 46]

    * [cite_start]약속 장소에서 서로를 찾지 못하는 문제를 해결할 전용 서비스의 부재 [cite: 48, 49]
* **활용방안**:
    * [cite_start]**방 개설 및 참여**: 출발지, 목적지, 시간 조건에 따른 방 생성 및 검색 [cite: 53, 54]

    * [cite_start]**실시간 위치 공유**: Kakao Map API를 연동하여 매칭 완료 후 탑승지에서 서로의 실시간 위치 확인 [cite: 56, 80, 81]

# 🚀 아키텍처 가이드 및 주의사항 (`feature/unified-server` 기준)

## 1. 브랜치 주요 변경사항
* **서버 통합 (단일 터널링)**: 기존에는 프론트엔드(Vite)와 백엔드(FastAPI) 터널을 각각 띄워야 했으나, 본 브랜치에서는 프론트엔드를 빌드(`dist`)하여 FastAPI 백엔드가 정적 파일로 한 번에 제공하도록 아키텍처를 통합했습니다.
* **API 상대 경로 전환**: 프론트엔드의 모든 통신 주소를 절대 경로에서 상대 경로(`/api/...`)로 변경하여, 접속 도메인이 바뀌어도 코드를 수정할 필요가 없도록 유연성을 확보했습니다.
* **502 Bad Gateway 원천 차단**: 서버가 통합되어 프론트엔드 모듈이 단일 빌드 파일로 압축 제공되므로, 무료 터널링 서버가 감당하지 못해 발생하던 통신 에러와 CORS 문제를 근본적으로 해결했습니다.

## 2. AWS 올릴 때 고려해야 하는 필수 설정
실제 클라우드(EC2) 환경에 배포하기 위해서는 인프라 구성 후 아래의 보안 및 연결 설정을 반드시 진행해야 합니다.

* **`.env` (보안 환경 변수) 세팅**
  * 카카오 API 키 등의 보안 정보는 깃허브에 절대 올라가지 않도록 제외되어 있습니다.
  * AWS 배포 시 EC2 서버의 `frontend/` 폴더 내부에 `.env` 파일을 수동으로 생성하고, 발급받은 **'카카오 JavaScript 키'**를 `VITE_KAKAO_MAP_KEY`로 입력한 뒤 반드시 `npm run build`를 수행해야 지도가 정상 작동합니다.
* **AWS RDS (데이터베이스) 구성**
  * 로컬 개발용 임시 DB(SQLite)를 버리고, 생성한 RDS의 엔드포인트로 변경해야 합니다.
  * **수정 위치**: `backend/main.py` (상단 `DATABASE_URL` 변수)
  * `DATABASE_URL = "postgresql://<아이디>:<비밀번호>@<AWS-RDS-주소>:5432/<DB이름>"`
* **AWS ElastiCache (Redis) 구성**
  * 실시간 위치 웹소켓 통신을 관리하는 로컬 Redis 주소를 ElastiCache 엔드포인트로 변경해야 합니다.
  * **수정 위치**: `backend/main.py` (`redis_client` 변수)
  * `redis_client = redis.from_url("redis://<AWS-ElastiCache-주소>:6379", decode_responses=True)`

## 3. 사용할 때 주의할 점

### 💡 API만 테스트할 때 (혼자 로컬 개발 시)
* 터널링 프로그램이 전혀 필요 없습니다. Redis 서버(`redis-server`)와 백엔드 서버(`uvicorn main:app --reload`)만 켜고 브라우저에서 `http://127.0.0.1:8000` 으로 접속하면 전체 앱 테스트가 가능합니다.
* 단, 카카오 개발자 사이트 'Web 플랫폼' 도메인 설정에 `http://127.0.0.1:8000` 과 `http://localhost:5173` 이 미리 등록되어 있어야 로그인이 작동합니다.

### 💡 실제 터널링할 때 필요한 절차 (원격 팀원 테스트 시)
* 프론트엔드용 터널은 절대 켜지 않으며, **백엔드 서버용 터널 딱 1개**만 실행합니다.
  ```bash
  npx localtunnel --port 8000 --subdomain <원하는이름>
  ```
* 터널을 켤 때 발급받은 주소(예: `https://...loca.lt`)를 카카오 개발자 사이트의 'Web 플랫폼' 도메인에 수동으로 갱신해 주어야 로그인이 막히지 않습니다.
* `backend/main.py` 의 `allow_origins` 리스트에도 해당 터널 주소가 추가되어 있어야 정상적으로 통신이 허용됩니다.

### 💡 AWS 배포 시 수정해야 하는 부분 (터널링 찌꺼기 제거)
* 로컬터널(`loca.lt`) 주소는 임시 테스트용이므로, 클라우드 파트가 AWS 배포를 완료하여 공인 IP/도메인이 생기면 기존 터널링 관련 설정들을 깨끗하게 지워야 합니다.
* `backend/main.py` 의 `allow_origins` 리스트에서 `.loca.lt` 로컬터널 주소를 지우고, **AWS의 실제 퍼블릭 IP나 구매한 도메인 주소**로 교체해야 합니다.
* 카카오 개발자 사이트 'Web 플랫폼' 설정 역시 로컬터널 주소를 삭제하고, **AWS 도메인**으로 교체해야 에러가 발생하지 않습니다.
